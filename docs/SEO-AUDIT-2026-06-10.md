# SEO Audit & Improvement Plan — bitcoinweighin.com

*Audit date: 2026-06-10. Analysis only; no code modified. Code citations verified against the working tree at commit `e654fbc`; live-site claims verified by fetching production on 2026-06-10. Companion to `docs/AUDIT-2026-06-09.md` (repo audit).*

---

## 1. Executive Summary

Overall SEO health: **B for on-page, D for off-page/infrastructure — blended C+**. The on-page work is well above the norm for a solo project: every route is fully prerendered with real content and live numbers baked into the HTML, every page has a unique title, meta description and canonical, the structured data (WebSite, Organization, Dataset, FAQPage, BreadcrumbList) is correct and mirrors visible content, `robots.txt` and `llms.txt` are exemplary, and the Core Web Vitals posture (LCP preload, inlined critical CSS, reserved heights) is deliberate and good. What undermines it sits one layer down: **the site appears to not be indexed at all** (a search for the domain returns nothing, and the repo contains no trace of Search Console verification or sitemap submission); **www serves a full 200 duplicate of every page** with no host redirect; and **every non-existent URL serves the homepage instead of a 404** (Cloudflare Pages SPA fallback), a textbook soft-404 factory. Top 3 risks: (1) Google may simply never have been told the site exists — all on-page quality is moot until then; (2) duplicate-host and soft-404 behaviour waste crawl budget and dilute signals exactly when the site is trying to establish itself; (3) the hand-maintained `sitemap.xml` will silently drift from the route set (the same drift class the repo audit found in pricing constants). Top 3 opportunities: (1) Search Console + Bing verification and sitemap submission — an afternoon's work that unblocks everything; (2) host canonicalisation and a real 404 — two Cloudflare-level fixes; (3) the dataset/data page is genuinely link-worthy (CC-BY, checksums, citation formats, Dataset JSON-LD) and is the natural magnet for the only durable ranking input: links from analysts, journalists and notebooks.

## 2. SEO Surface Map

**Indexable routes (22 URLs, all prerendered):** `/` (interactive visualiser + long-form SEO section + FAQ), `/data` (dataset landing page), `/methodology`, `/btc/{gold,silver,pu238,cocaine}` (commodity landing pages), `/snapshot` (index), `/snapshot/2013..2026` (14 year pages). All emitted as static HTML by `adapter-static`; all carry self-referencing canonicals to the apex domain.

**Non-HTML SEO surfaces:**
- `static/robots.txt` — allow-all with explicit grants for 20+ search and AI crawlers, sitemap pointer (`robots.txt:91`).
- `static/llms.txt` — full site description, primary pages, machine-readable endpoints, licensing/citation. Unusually good; most sites don't have one.
- `static/sitemap.xml` — hand-maintained, 22 `<loc>` entries, no `<lastmod>` anywhere (`grep -c lastmod` = 0).
- `functions/og-image.ts` — dynamic 1200×630 PNG share cards (`og-image.ts:381`), 1-hour edge cache (`og-image.ts:284–285`).
- `static/_headers` — edge cache rules for HTML on `/`, `/data`, `/methodology` only (`_headers:53–58`).

**SEO code:** `src/lib/seo/` — `jsonld.ts` (five JSON-LD builders, `<` escaped at line 15), `commodity-content.ts` (per-commodity titles/copy/FAQs with `{ratio}` substitution at build time), `faqs.ts` (homepage FAQ, single source for page + JSON-LD), `snapshots.ts` (year summaries derived from `prices.json`).

**Daily freshness loop:** the 02:00 UTC price workflow redeploys the site, so prerendered pages re-bake with the new ratio — titles stay keyword-stable while body copy and FAQ answers update (deliberate, documented at `+page.svelte:228–231` and `btc/[commodity]/+page.server.ts:69–74`). This is the site's structural advantage: "fresh daily numbers in static HTML" is something most competitors in this query space don't do.

## 3. Audit Report

Severity scale: Critical / High / Medium / Low. Each finding labelled **[fact]** or **[judgment]**.

### 3.1 Indexation & discovery

**I1 — HIGH [fact + judgment]. No detectable search presence, and no evidence the site has ever been submitted to a search engine.**
A web search for `bitcoinweighin.com` (2026-06-10, US index) returns no result for the domain on the first page — only generic bitcoin-price sites. The repo contains no Search Console or Bing verification artifact (no verification meta tag in `src/app.html`, no `google*.html` in `static/`, no mention in `docs/`). Cloudflare DNS-level verification can't be ruled out from here **[judgment]**, but the combination of a recently launched site, zero inbound links (GitHub repo: 0 stars/forks), and a sitemap that has never demonstrably been submitted means the most probable state is: Google has not crawled the site, or has crawled and not indexed it. Everything else in this audit is secondary to resolving this.

**I2 — HIGH [fact]. `www.bitcoinweighin.com` serves the entire site as a 200 duplicate with no redirect to the apex.**
Fetching `https://www.bitcoinweighin.com/` returns the full homepage on the www host (no redirect; final URL stays www). Stranger: fetching apex `/data/` (trailing slash) lands on `https://www.bitcoinweighin.com/data` — Cloudflare Pages' slash-normalisation redirect hops *across hosts*, suggesting www is the Pages project's primary custom domain while all canonicals, the sitemap, `robots.txt` and `llms.txt` declare the apex. Canonical tags point to apex on every page, which mitigates indexation damage, but canonicals are hints, not directives; a crawler discovering the site via www (or via that cross-host redirect) sees a parallel 22-page site. One 301 redirect rule fixes the whole class.

**I3 — HIGH [fact]. Every non-existent URL serves the homepage — soft-404s sitewide.**
`https://bitcoinweighin.com/this-page-does-not-exist-xyz` returns the complete homepage document (title, homepage meta, homepage JSON-LD) at the requested URL. Cause: `svelte.config.js:11` sets `fallback: undefined`, so the build emits no `404.html`; in that state Cloudflare Pages enables SPA mode and serves `index.html` for unmatched paths. There is also no `src/routes/+error.svelte`. Consequences: any typo'd, hallucinated (LLMs will invent `/btc/oil`), or legacy URL returns 200 with duplicate homepage content; crawl budget leaks; Search Console (once set up) will fill with "Duplicate, Google chose different canonical" noise. The homepage canonical on those responses limits the damage but doesn't fix the wrong status code.

**I4 — MEDIUM [fact]. `sitemap.xml` is a hand-maintained static file with no `<lastmod>`, guaranteed to drift.**
No script in `scripts/` or `src/` generates it (grep for "sitemap" across both: no hits). The snapshot year pages, by contrast, are *generated from data* (`snapshot/[year]/+page.server.ts:13–15` derives `entries` from `listYears()`), so on 2027-01-01 a `/snapshot/2027` page will exist that the sitemap doesn't list — the exact "two sources of truth, no test guarding the seam" failure mode the repo audit catalogued for prices. `<lastmod>` is the one sitemap field Google actually uses (it largely ignores `<changefreq>`/`<priority>`, both of which this file maintains carefully); for a site whose pitch is "updated daily", omitting it discards a real crawl-scheduling signal.

### 3.2 Duplication & canonicalisation

**C1 — Strength [fact].** Every indexable page carries a correct self-referencing canonical to the apex host (verified live on `/`, `/btc/gold`, `/snapshot/2021`, `/data`). Homepage parameter states (`/?btc=…&date=…&commodity=…`) all canonicalise to `/` — the right call for a URL-as-state design.

**C2 — LOW [fact]. The baked `og:url` on the homepage is a dated parameter URL that changes every day.**
The prerendered HTML ships `og:url: https://bitcoinweighin.com/?date=2026-06-08` and `og:image: /og-image?btc=1&date=2026-06-08` (verified live), because the SSR pass evaluates the reactive builders at `+page.svelte:250–267` with the build-day state. `og:url` should equal the canonical for the default state; the daily-shifting URL fragments the share/link graph (each day's homepage shares point at a different parameterised URL) and the dated og:image defeats the share-card's "today" framing for anyone sharing the bare homepage a week later. Trivial fix: emit canonical `/` and an undated og:image in the default state, keep the dynamic values only after user interaction.

### 3.3 Metadata & social surfaces

**M1 — Strength [fact].** Titles and descriptions are unique, well-written and intent-matched across all 22 pages: commodity pages lead with the question ("Bitcoin to Gold: How Much Gold Does 1 BTC Buy Today?" — `commodity-content.ts:37`), snapshot pages target year queries ("Bitcoin in 2021: BTC → Gold, Silver & USD Through the Year"), the data page leads with what the asset is. Description lengths are within range. FAQ copy is substituted with the live ratio at build time so visible content and JSON-LD match verbatim (`btc/[commodity]/+page.server.ts:69–80`).

**M2 — MEDIUM [fact]. Open Graph / Twitter coverage is incomplete on exactly the most shareable pages.**
`/data` and `/methodology` have *no* `og:*` or `twitter:*` tags at all (`data/+page.svelte:61–111`, `methodology/+page.svelte:16–25` — head blocks contain title/description/canonical/JSON-LD only). Snapshot pages have og:title/description/url but no `og:image` and no `twitter:card` (`snapshot/[year]/+page.svelte:50–55`, `snapshot/+page.svelte:35–39`). The dataset page is the page most likely to be linked by analysts and journalists; shared today it renders as a bare-text card. The OG endpoint already exists and takes a `date` param — snapshot pages could even pass their year-end date for a period-correct card.

**M3 — LOW [judgment]. The homepage title spends its best characters on the brand and promises a commodity that has no page.**
`Bitcoin Weigh-In — BTC to Gold, Silver, Oil & Commodities Visualised` (`+page.svelte:233–236`, 69 chars — will truncate in most SERPs). Brand-first ordering wastes the highest-weight slot on a brand nobody searches for yet; "Oil" appears in the title, the meta description and the visible subtitle (`+page.svelte:411–413`) but `/btc/oil` doesn't exist (oil is in the dataset, not the visualisation). Either ship the oil landing page (the dataset already has `brent_per_btc` daily history) or stop advertising it in the three most prominent strings on the site.

**M4 — LOW [fact].** `rel=prev/next` on snapshot pages (`snapshot/[year]/+page.svelte:43–48`) — Google retired these as indexing signals in 2019. Harmless; zero maintenance cost; note only.

### 3.4 Structured data

**S1 — Strength [fact].** The JSON-LD layer is correct and disciplined: `<` escaped against script injection (`jsonld.ts:15`); Dataset markup on `/data` with distributions, temporalCoverage, license URL and conditional DOI (`data/+page.svelte:74–108`) — eligible for Google Dataset Search, which is a real discovery channel for this asset; Organization/WebSite with `@id` cross-references; BreadcrumbList matching visible breadcrumbs; FAQPage matching visible `<dl>` content verbatim.

**S2 — LOW [judgment]. FAQPage markup will not produce rich results.** Since Google's 2023 restriction, FAQ rich snippets are limited to authoritative government/health sites. The markup still earns its keep as a machine-readable Q&A surface for LLM crawlers (which this site explicitly courts in `robots.txt`), so keep it — just don't expect SERP real estate from it.

**S3 — LOW [fact].** Two trivia: the `jsonld.ts` header comment (line 7) claims a `SearchAction` that is never emitted (and the site has no search — fix the comment, not the markup); the Organization `logo` (`jsonld.ts:25`) points at `header@1x.webp`, a wide banner, where Google's guidance wants a roughly square image ≥112×112px.

### 3.5 Content & internal architecture

**A1 — Strength [judgment].** The content strategy is sound and honestly executed: each commodity page targets a clear question cluster ("how much gold does 1 bitcoin buy", "btc to gold ratio") with the answer in the first sentence, baked into static HTML, refreshed daily; snapshot pages own the long-tail "bitcoin in {year}" space with month-by-month tables no competitor renders statically; the homepage carries a genuine long-form section rather than hidden SEO text. Illustrative-price disclaimers (pu238, cocaine) are visible on-page and consistent with the methodology page — good E-E-A-T hygiene, as are the checksums, citation formats, provenance table and public repo on `/data`.

**A2 — MEDIUM [fact]. Internal linking is hub-and-spoke; the spokes barely link to each other.**
`/data` and `/methodology` contain zero links to any `/btc/*` or `/snapshot*` page (grep across both `+page.svelte` files: 0 matches). Commodity pages link to sibling commodities and home, but not to any snapshot page; snapshot year pages link to `/btc/gold` only in a footer sentence. The homepage does all the lifting. For 22 pages this won't strand anything, but it forfeits the easiest relevance signal available: `/btc/gold` ↔ "bitcoin in 2021" (when gold context changed most), `/data` → the pages that visualise each column.

**A3 — LOW [judgment]. The dataset has seven commodities with daily history; the site has landing pages for four.**
`brent_per_btc`, `xpt_per_btc`, `copper_per_btc`, `wheat_per_btc`, `coffee_per_btc` columns exist back to 2013, and `commodity-content.ts` is structured to take new entries cheaply. "Bitcoin to oil/platinum/copper" pages are low-competition queries the data already answers; the homepage title already promises oil (M3). Counterweight: the repo audit's open question #4 notes deferred commodities are a conscious scope decision — so this is a content roadmap item, not a defect.

### 3.6 Performance & crawl efficiency

**P1 — Strength [fact].** The CWV work is real: LCP sprite preloaded with `fetchpriority=high` and srcset (`app.html:22–27`), self-hosted subset fonts preloaded, critical CSS inlined post-build, reserved heights against CLS, latest-day prices inlined so the prerendered HTML paints meaningful content without JS. Crawlers that don't execute JS get the full content — verified live.

**P2 — LOW [fact]. The edge-cache rules that exist *because* origin-served HTML measurably regresses TTFB don't cover 18 of the 22 pages.**
`static/_headers:53–58` applies `s-maxage=86400` to `/`, `/data`, `/methodology` only — the file's own comment explains uncached HTML goes `cf-cache-status: DYNAMIC` with a "measurable TTFB regression on the LCP path". `/btc/*` and `/snapshot*` — the dedicated SEO landing pages — are exactly the pages left on the slow path. Two glob lines fix it.

**P3 — LOW [judgment].** Two analytics scripts load on every page (Cloudflare beacon + Umami, `app.html:28–31`). Both are lightweight and deferred; consolidating to one is a nicety, not a need.

**P4 — info [fact].** Live data was one day behind at audit time: dataset coverage ended 2026-06-08 with the `/data` page showing "Generated 2026-06-09", fetched on 2026-06-10 — the 02:00 UTC run on the 10th had either not landed or not deployed. Single observation, not a trend; worth a glance at `health.json`/Actions. The freshness loop is the SEO moat — a silent multi-day stall would quietly rot every "today" claim on the site.

### 3.7 Off-page

**O1 — HIGH [fact + judgment]. Effectively zero inbound links.** The GitHub repo (the only known external link source) has 0 stars and 0 forks; the domain returns nothing in search. On-page quality cannot rank a site with no links in a competitive query space ("bitcoin to gold" is contested by calculators and exchanges). The CC-BY dataset with checksums, versioning, DOI-pending Zenodo archive and citation formats is the natural link asset — datasets earn links that visualisations don't. The planned Zenodo DOI (`docs/zenodo-release.md`, staged) and Google Dataset Search via S1 are the two highest-leverage moves already in flight; the X bot posting 5×/day creates discovery but X links are nofollow — they seed humans, not PageRank **[judgment]**.

### 3.8 Strengths (what to preserve)

Full prerendering with live numbers in static HTML and a daily redeploy loop; keyword-stable titles with dynamic share surfaces (the documented title-churn lesson at `+page.svelte:228–231`); the `robots.txt` AI-crawler policy and `llms.txt` (ahead of the curve, aligned with the CC-BY positioning); Dataset JSON-LD with verbatim-matching visible content; canonical discipline across parameter states; alt-text discipline (decorative `alt=""` vs descriptive alts with real dimensions); one `<h1>` per page with sane heading hierarchy; E-E-A-T surfaces (methodology, provenance, checksums, corrections contact, public repo).

## 4. Improvement Strategy

Four themes explain nearly every finding.

**Theme 1 — Exist, first (I1, O1).** Nothing else in this document matters until a search engine knows the site exists and a human can verify what it sees. *Target state:* Search Console and Bing Webmaster Tools verified, sitemap submitted, all 22 pages requested for indexing, index coverage checked weekly until stable. *Principle:* on-page SEO is multiplication; indexation is the non-zero term.

**Theme 2 — One host, honest status codes (I2, I3, C2).** The site currently tells crawlers three small lies: www is a separate 200 site, missing pages are the homepage, and the homepage's share URL is a dated variant of itself. *Target state:* www 301s to apex at the edge; unknown paths return HTTP 404 with a real error page; default-state og:url equals the canonical. All three are configuration, not code.

**Theme 3 — Generated artifacts, not hand-maintained ones (I4, M2).** The sitemap should be derived from the same route definitions that generate the pages — the repo audit's Theme 1 ("duplication is acceptable only when a test makes silent drift impossible") applies verbatim; a build-time sitemap with `<lastmod>` from the dataset's `last_updated` removes the drift class and adds the one signal Google reads. Same spirit for OG tags: one shared head component/snippet so a new route can't ship half a share surface.

**Theme 4 — Calibrated restraint (what NOT to do).** Not recommended at this maturity: no blog/content-marketing treadmill (the dataset and bot are the channel); no hreflang (single-language site); no further CWV tuning (already past the point of diminishing returns); don't remove FAQPage markup despite no rich results (it serves the AI-crawler strategy); don't robots-block `/og-image` (the unbounded query space is theoretical — parameter URLs all canonicalise, and blocking could interfere with image fetching); don't chase `rel=prev/next` removal or other zero-cost archaisms.

**Definition of done — measurable signals:**
- Site appears in Google/Bing for `bitcoinweighin` and `site:bitcoinweighin.com` shows ~22 pages.
- `curl -sI https://www.bitcoinweighin.com/` → `301` to apex; nonsense URL → `404`.
- Sitemap generated at build with `<lastmod>`; zero hand-edited URL lists in `static/`.
- Every indexable page ships og:title/description/url/image + twitter:card (one shared component).
- Search Console: zero soft-404s, zero "duplicate without user-selected canonical" on www.
